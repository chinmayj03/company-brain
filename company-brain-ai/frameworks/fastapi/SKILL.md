You are extracting from a **FastAPI** codebase. Apply these conventions
when picking entities, edges, and call chains for the brain store.

## Routing & entry-point patterns

- `app = FastAPI(...)` at module level is the application root. Routers are
  `router = APIRouter(prefix="/foo", tags=[...])` modules included via
  `app.include_router(router)`.
- Decorators `@app.get("/path")`, `@app.post(...)`, `@router.get(...)`, etc.
  mark entry-point coroutines. Capture the HTTP method, the full path
  (router prefix + decorator path), and the handler function.
- Handlers are usually `async def` and return Pydantic models or `dict`s.
  Treat the handler as a function_node entry point.
- `dependencies=[Depends(...)]` (router or decorator level) is a middleware
  chain — record each `Depends` callable as a DEPENDS_ON edge from the
  handler.
- `BackgroundTasks` parameters mean post-response work; capture the
  `tasks.add_task(fn, ...)` callee as a CALLS edge.

## Pydantic models to skip (do NOT extract methods on these)

- `class FooRequest(BaseModel)`, `class FooResponse(BaseModel)`, and
  `*In` / `*Out` schema classes whose only methods are validators or
  Pydantic config. Capture the type *name* on edges; skip the methods.
- `*Settings` / `BaseSettings` classes — config, not business logic.
- `*Enum(str, Enum)` value classes.

If a Pydantic model has a real method (custom serializer, computed field,
business helper), extract that one method only.

## Database access patterns

- **SQLAlchemy ORM**: a session is usually injected via `Depends(get_db)`.
  Look for `db.query(Model).filter(...).first()` / `.all()` chains, or
  the 2.0-style `select(Model).where(...)` + `db.execute(stmt).scalars()`.
  Capture each `Model` as a DEPENDS_ON target and the raw column predicates
  as `query_text`.
- **SQLAlchemy Core**: `Table` objects + `select()` / `insert()` / `update()`.
  Walk the `.from_(...)` / `.where(...)` chain.
- **Async drivers**: `await db.execute(stmt)`; treat the same as sync.
- **Raw SQL**: `db.execute(text("SELECT ..."))` — capture the SQL string
  verbatim as `query_text`.
- **Repositories**: many FastAPI projects layer a `class FooRepository`
  over the session. Methods on that class are data-access entry points
  worth extracting in full.

## Service-layer conventions

- Services are usually plain classes injected via `Depends(get_foo_service)`.
  The handler calls `service.do_thing(...)`; record CALLS edges accordingly.
- Functional alternatives: a module-level `def do_thing(...)` called directly
  from the handler. Same edge shape, different node type.

## Common false positives

- `Depends(...)` is a marker, not a call site. The CALL is from the handler
  to the dependency callable, not to `Depends` itself.
- `Field(...)`, `Query(...)`, `Path(...)`, `Body(...)` are parameter
  declarations; ignore them as call targets.
- `HTTPException(status_code=...)` is a raise, not a CALLS edge worth
  recording.
- `JSONResponse(...)`, `Response(...)` wrappers — capture the wrapped
  payload type.
- `model.dict()`, `model.model_dump()`, `model.json()` are Pydantic
  serialisation; not real CALLS edges to chase.

## DTOs that look like business logic but aren't

- `*Schema` classes with `from_orm` / `model_validate` only.
- Any class generated by `pydantic.create_model(...)`.

## When in doubt

- Handler → service → repository is the canonical chain. Stop at
  third-party-library boundaries (e.g. don't extract methods on
  `sqlalchemy.orm.Session`).
- Prefer one batched `extract_methods_from_class` per file over many
  per-method calls.
